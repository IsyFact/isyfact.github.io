Neue Dokumente anlegen

Diese Seite ist ein Teil der IsyFact-Standards. Alle Inhalte der Seite, insbesondere Texte und Grafiken, sind urheberrechtlich geschützt. Alle urheberrechtlichen Nutzungs- und Verwertungsrechte liegen beim Bundesverwaltungsamt.

Die Nutzung ist unter den Lizenzbedingungen der Creative Commons Namensnennung 4.0 International gestattet.

In diesem Kapitel wird für die Zentrale Dokumentation gezeigt, wie die erforderliche Verzeichnisstruktur aussieht und wie ein neues Dokument dort einsortiert wird.

1. Zentrale Dokumentation: Ein neues Verzeichnis anlegen

Die zentrale Dokumentation der IsyFact-Standards gliedert sich nach den Säulen der IsyFact.

Allgemein

Einstiegsdokumente, Tutorial, Konventionen

Blaupausen

Grundlagendokumente und Detailkonzepte für Architektur und Anwendungslandschaft

Bausteine

Dokumente für die jeweiligen wiederverwendbaren Softwarelösungen

Methodik

Methodische Dokumente bspw. zu Programmierkonventionen und Systemhandbuch und -entwurf

Werkzeuge

Dokumente zur Verwendung von Werkzeugen und der Einrichtung der Entwicklungsumgebung (hierzu gehört auch dieses Dokument)

Ein neues IsyFact-Dokument muss unterhalb einer geeigneten Säule abgelegt werden. Dazu wird unterhalb des Verzeichnisses, das die Säule repräsentiert, ein neues Verzeichnis angelegt, dessen Name den Inhalt des Dokuments widerspiegelt.

1.1. Dokumentation eines Bausteins: Ein neues Verzeichnis anlegen

Soll für einen Baustein eine neue Dokumentation angelegt werden oder die bestehende Dokumentation um ein Dokument erweitert werden, so muss zunächst der geeignete Ort zur Ablage bestimmt werden. I.d.R. ist dies das Repository, in dem sich der Baustein befindet, da die Dokumentation nah beim Code abgelegt werden soll. Früher befand sich jedoch ein großer Teil der Bausteindokumentation im Repository isy-documentation.

Aktuell liegt dieser Teil der Bausteindokumentation im Repository isyfact-standards im Submodul isyfact-standards-doc. Der Code dieser Bausteine befindet sich ebenfalls im Repository isyfact-standards, wodurch die Architekturregel "Dokumentation nah beim Code" erfüllt ist.

Zunächst muss der Baustein vom Git-Repository heruntergeladen werden. Hierfür muss auf der Kommandozeile auf eine geeignete Stelle der lokalen Festplatte gewechselt werden und folgendes Kommando eingegeben werden.

C:\tmp\> git clone https://github.com/IsyFact/<Repository>;

z.B. C:\tmp\> git clone https://github.com/IsyFact/isy-baustein

Bei <Repository> kann es sich um einen einzelnen Baustein handeln oder um ein Repository mit zentraler Dokumentation wie z.B. isyfact-standards.

In einem Baustein wird die Dokumentation in einem eigenen Maven-Submodul unter dem Pfad src/docs abgelegt.

Verzeichnisse der Bausteindokumentation zeigt die Verzeichnisstruktur für die Dokumentation des fiktiven Bausteins isy-baustein zusammen mit den AsciiDoc-Dateien für ein Dokument.

Listing 1. Verzeichnisse der Bausteindokumentation
📂 isy-baustein
  📂 isy-baustein-doc (1)
    📂 src
        📂 docs
            📂 Konzept (2)
                📄 anhaenge.adoc  (3)
                📄 docinfo.adoc
                📄 inhalt.adoc    (4)
                📄 master.adoc    (5)
                📄 thisdoc.adoc   (6)
                📂 images (7)
            📂 Nutzungsvorgaben (2)
                📄 anhaenge.adoc
                📄 docinfo.adoc
                📄 inhalt.adoc
                📄 master.adoc
                📄 thisdoc.adoc
                📂 images
1 Maven-Submodul für die Dokumentation.
2 Jeweils ein Dokument; bei Bausteinen findet sich häufig die Unterteilung in Konzept und Nutzungsvorgaben.
3 Auflistung der Anhänge des Dokuments.
4 Inhalte des Dokuments.
5 Zentrale Datei, die alle anderen Dateien über spezielle Verweise (include) einbindet.
6 Einleitendes Kapitel des Dokuments, das den Inhalt zusammenfasst sowie den Aufbau und Zweck des Dokuments erläutert.
7 Bilder und Grafiken, die im neu zu erstellenden Dokument verwendet werden sollen.
Eine vollständige Vorschau des Dokuments wird angezeigt, wenn die Datei master.adoc im Editor geöffnet und die Vorschaufunktion verwendet wird.

2. Struktur eines Dokuments

Wie bereits im vorigen Abschnitt beschrieben, besteht ein Dokument aus mehreren Dateien. Die folgenden Abschnitte beschreiben diese Dateien, stellen Vorlagen bereit und erklären Einstellungsmöglichkeiten in Form von Attributen.

2.1. master.adoc

Die Datei master.adoc fasst alle anderen Dateien zu einem Dokument zusammen. Die folgende Vorlage zeigt den Aufbau der Datei.

Listing 2. Vorlage für master.adoc
:doctitle: Neuer Baustein
:document-type: Konzept
:vs-einstufung: offen

// :revnumber: 1.0.5
// :revdate: 2020-01-01

colophon::[factory="IsyFact-Erweiterungen", open-source="false"]

include::docinfo.adoc[]

include::thisdoc.adoc[tags=inhalt]

include::inhalt.adoc[tags=inhalt]

:sectnums!:

include::anhaenge.adoc[tags=inhalt]

Am Anfang der Datei werden die Attribute doctitle und document-type gesetzt, aus denen ein einheitlicher Titel erzeugt wird.

Dann wird die Vertraulichkeitseinstufung vs-einstufung angegeben. Dadurch wird ggf. ein Vertraulichkeitshinweis in das Dokument eingefügt und ggf. das Layout angepasst. Die möglichen Werte für vs-einstufung finden sich in Übersicht der Attribute.

Die vs-einstufung kann auch im Maven-POM als globales Attribut für alle zu generierenden Dokumente angegeben werden.

Die manuelle Vergabe von Revisionsnummer und -datum betrifft nur Dokumente, die nicht mithilfe des Maven-Plugins erzeugt werden. Ansonsten werden diese Attribute über das Maven-Plugin bereitgestellt.

Die folgende Tabelle zeigt alle Attribute, die in der master.adoc gesetzt werden können.

Tabelle 1. Attribute der master.adoc
Attribut Optional Beschreibung Werte

doctitle

Nein

Titel des Dokuments

document-type

Ja

Dokumententyp (z.B. Konzept), welcher als Untertitel angezeigt wird

vs-einstufung

Nein

Vertraulichkeitseinstufung

  • open-source

  • offen

  • einstufung-fehlt

  • vs-nfd

Nach den Attributen wird der Kolophon eingebunden. Dadurch wird ein einheitlicher Abschnitt mit Metadaten des Dokuments und Lizenzbestimmungen erzeugt. Dabei können noch die optionalen Parameter factory und open-source angegeben werden.

Tabelle 2. Parameter des Kolophons
Parameter Optional Default Beschreibung Werte

Ja

  • IsyFact-Standards

  • IsyFact-Erweiterungen

  • Register Factory

open-source

Ja

true

Beschreibt, ob das Dokument unter der Open Source (true) oder Closed Source (false) Lizenz der IsyFact veröffentlicht wird.

Nach dem Kolophon werden die übrigen AsciiDoc-Dateien in master.adoc inkludiert. Leere Dateien können ausgelassen werden. Inkludiert wird der Inhaltsbereich innerhalb der Dateien, der mit entsprechenden Tags gekennzeichnet ist.

2.2. docinfo.adoc

Dokumente können sich direkt auf Bibliotheken oder IT-Systeme beziehen. Ist dies der Fall, so werden diese Bibliotheken oder IT-Systeme in einer Datei docinfo.adoc angegeben. Neben dem technischen Namen muss auch die Version angegeben werden, auf die sich das Dokument bezieht. Die docinfo.adoc Datei wird dann in master.adoc direkt nach dem Kolophon eingebunden.

Die folgende Vorlage zeigt den Aufbau der Datei.

Listing 3. Vorlage für docinfo.adoc
*Java Bibliothek / IT-System*

[cols="5m,2,3",options="header"]
|====
|Name |Art |Version
|<Technischer Name> |Bibliothek | <Version>
|<Technischer Name> |IT-System  | <Version>
|====

2.3. thisdoc.adoc

Die Datei thisdoc.adoc bietet eine Einleitung in das Dokument und erläutert den Aufbau und Zweck des Dokuments. Die folgende Vorlage zeigt den Aufbau der Datei.

Listing 4. Vorlage für thisdoc.adoc
= Konzept: Neuer Baustein - Einleitung
:imagesdir: images

// tag::inhalt[]

== Einleitung
// Einordnung des Dokuments

=== Aufbau und Zweck des Dokuments
// Informationen zur Zielgruppe des Dokuments
// Kurze Zusammenfassung des Aufbaus
// Auflistung der Kapitel und ihrer Kerninhalte

// end::inhalt[]

2.4. inhalt.adoc

In inhalt.adoc wird der eigentliche Inhalt des Dokuments verfasst. Die folgende Vorlage zeigt den Aufbau der Datei.

Listing 5. Vorlage für inhalt.adoc
= Konzept: Neuer Baustein - Inhalt
:imagesdir: images

// tag::inhalt[]

// Eigentlicher Inhalt des Dokuments

== Erstes Thema
...

== Zweites Thema
...

// end::inhalt[]

2.5. anhaenge.adoc

In anhaenge.adoc sind Anhänge zum Dokument platziert. Die folgende Vorlage zeigt den Aufbau der Datei.

Listing 6. Vorlage für anhaenge.adoc
= Konzept: Neuer Baustein - Anhänge
:imagesdir: images

// tag::inhalt[]

== Anhang A: Titel
// Inhalt des Anhangs

== Anhang B: Titel
// Inhalt des Anhangs

// end::inhalt[]
